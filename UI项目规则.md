# UI项目规则

本文件用于约束在本仓库内进行分析、回答、编码与评审时的默认行为。

## 1. 基础要求

- **语言**：所有回复、注释、文档说明统一使用**中文**。
- **优先级**：先遵守本文件，再遵守 `README.md` 中的开发规范；如两者冲突，以更严格者为准。
- **真实性**：回答和代码修改应基于当前仓库实际结构，不臆测未落地能力。
- **复用性**：修改时优先复用现有封装、目录结构、命名方式与组件体系，避免重复造轮子。

## 2. 项目技术基线

> 请根据实际技术栈调整以下列表
- **前端框架**：[Vite] + [Vue 3 / React] + [TypeScript]
- **状态管理**：[Pinia / Redux Toolkit]
- **UI 组件库**：[Element Plus / Ant Design / ShadcnUI]
- **请求库**：[Axios]，统一封装请求方法
- **样式方案**：[Sass/Less] + [UnoCSS/TailwindCSS]
- **图标方案**：统一使用封装后的图标组件，不直接使用 SVG 字符串
- **工具库**：优先使用 [Lodash-es], [Dayjs], [Type-fest] 等标准库

### 2.1 内网/安全部署约束

- **禁止外部依赖**：本项目默认部署在内网或受限环境，**禁止**向互联网发送任何非必要的资源请求。
- **资源本地化**：所有运行时依赖资源必须本地化，包括字体、图标、地图、第三方 SDK 静态文件等。
- **禁止 CDN**：禁止在代码中引用 `unpkg`、`jsdelivr`、`cdnjs`、Google Fonts 等在线 CDN 地址。
- **第三方库检查**：接入第三方库时，必须检查其是否默认请求外部资源（如遥测、埋点），如有必须关闭或替换。

## 3. 编码规则

### 3.1 通用规范

- **命名语言**：变量名、函数名、类型名、文件名片使用**英文**（PascalCase 或 camelCase），严禁使用拼音。
- **注释语言**：代码注释、文档说明使用**简体中文**。
- **类型安全**：
    - 避免使用 `any`，优先使用明确类型。
    - 无法确定类型时使用 `unknown` 并进行类型守卫。
    - 接口响应、函数参数、返回值必须显式声明类型。

### 3.2 框架特定规范 (以 Vue 为例)

- **组件写法**：统一使用 `<script setup lang="ts">`。
- **代码组织**：组件内部优先按“Props -> Emits -> 状态 -> 计算属性 -> 方法 -> 生命周期”顺序组织。
- **响应式**：派生数据放入 `computed`，避免模板中重复计算。
- **列表渲染**：必须提供稳定的 `key`。

### 3.3 请求与错误处理

- **统一入口**：所有 HTTP 请求必须从统一的 API 封装文件（如 `src/api/request.ts`）导入。
- **类型约束**：请求返回值必须通过泛型约束类型。
- **错误处理**：
    - 所有异步请求必须有 `try/catch` 或统一的错误拦截。
    - 业务页面优先复用统一的错误提示组件（如 `MessageError`）。
    - `catch` 块中按 `unknown` 处理错误，不假设错误结构。

## 4. 国际化规则

- **强制接入**：所有用户可见的文案（按钮、标签、提示、弹框）必须接入国际化系统（如 `vue-i18n`）。
- **禁止硬编码**：禁止在代码中直接硬编码展示文案。
- **Key 命名**：国际化 Key 采用层级命名法（如 `common.confirm`, `user.login.title`）。
- **多语言同步**：新增文案时，需同步补齐所有支持的语言包（如 zh-CN, en-US）。

## 5. UI 与交互规则

### 5.1 通用设计原则

- **控制台风格**：本项目定位为“企业级数据控制台”或“管理系统”。
    - 视觉语言应保持克制、工具化、结构化。
    - 背景以浅色、灰白为主，强调层级边界。
    - 避免使用大面积高饱和色块、霓虹特效或营销风格的 Hero Banner。
- **布局范式**：
    - 优先使用“顶部导航/工具栏 + 侧边栏 + 内容区”的经典布局。
    - 列表页优先采用“筛选区 + 表格/卡片列表”模式。
    - 设计器页面优先采用“左侧结构 + 中间画布 + 右侧属性”模式。

### 5.2 组件使用规范

- **图标**：统一使用封装的图标组件（如 `PbIcon` 或 `Icon`），禁止散落使用 SVG。
- **消息反馈**：
    - 成功/警告/错误消息统一使用全局消息组件（如 `ElMessage` 或 `Message.ts`）。
    - 危险操作必须有二次确认（Confirm Dialog）。
- **表单**：
    - 复杂表单优先使用成熟的校验库（如 `vee-validate` 或 `zod`）。
    - 按钮操作区统一右对齐，主操作按钮使用主色调。

## 6. 目录与命名

### 6.1 目录结构建议

应尽量遵循以下目录职责（根据实际项目调整）：

- `src/api/`：API 接口定义与请求封装
- `src/components/`：通用业务组件与基础组件
- `src/constants/`：常量定义
- `src/hooks/` 或 `src/composables/`：组合式函数
- `src/i18n/`：国际化配置
- `src/router/`：路由定义
- `src/stores/`：全局状态管理
- `src/types/`：TypeScript 类型定义
- `src/utils/`：工具函数
- `src/views/`：页面级视图

### 6.2 命名规范

- **文件命名**：
    - 页面组件：`PascalCase + View`（如 `UserListView.vue`）
    - 通用组件：`PascalCase`（如 `UserCard.vue`）
    - 工具/配置：`kebab-case` 或 `camelCase`
- **业务术语**：
    - 在代码中保持业务术语的一致性（例如：如果是“用户”，就不要混用“会员”、“账号”）。
    - 若涉及特定业务领域（如电商、金融），请在下方补充术语表。

## 7. 状态与存储

- **应用状态**：全局共享状态优先使用状态管理库（如 `Pinia`）。
- **本地持久化**：
    - 用户偏好、筛选条件等本地配置优先使用 `localStorage` 或 `sessionStorage`。
    - 建议封装统一的 Storage 工具，不直接调用 `window.localStorage`。
- **Key 管理**：存储键名统一定义在常量文件中，避免魔法字符串。

## 8. 交付与评审要求

- **完整性检查**：修改代码时，优先补齐对应的类型定义、国际化文案、错误处理和必要的导入。
- **安全检查**：若新增功能，默认检查是否存在外部网络请求、敏感信息泄露风险。
- **代码评审**：优先关注行为回归、类型安全、国际化遗漏、组件封装被绕开等问题。
- **说明变更**：说明变更时，优先给出受影响文件、核心原因、验证结果与剩余风险。
